# NWC Wallet Connection

Warikan Sats supports one wallet connection mode: Nostr Wallet Connect. The app
is a client-only application. It must not require, call, or depend on an
operator-run backend for wallet access.

## Scope

Supported:

- User-provided `nostr+walletconnect://` connection strings.
- Invoice creation through `make_invoice`.
- Invoice settlement lookup through `lookup_invoice`.
- Wallet balance reads through `get_balance`.
- Optional `payment_received` notifications when the wallet service advertises
  support for them.

Out of scope:

- WebLN.
- Lightning Address or LNURL-pay.
- Direct LND, Core Lightning, LNbits, or other REST APIs.
- Operator-managed wallet credentials or relay services.

The protocol reference is NIP-47:

- <https://nips.nostr.com/47>

## Connection Data

The user supplies an NWC URI generated by their wallet. The client must parse:

- wallet service public key from the URI host/path.
- one or more `relay` query parameters.
- the `secret` query parameter.
- optional metadata such as `lud16`.

Rules:

- Store connection data only in client-controlled storage.
- Never write connection strings, secrets, payment preimages, or wallet
  credentials to tracked files, logs, issue text, or telemetry.
- Treat the NWC `secret` as a wallet credential.
- Provide a clear disconnect action that removes locally stored connection
  data.
- Use a separate connection per app install or browser profile; do not ask for
  or reuse a user's primary Nostr identity key.

## Required Capabilities

Before enabling split-payment creation, the client should fetch the NWC info
event and require these capabilities:

- `make_invoice`
- `lookup_invoice`

`get_balance` is optional for the core split-payment flow, but the app should
use it when available for wallet overview data. If `get_balance` is missing,
the wallet overview should show a capability-specific unavailable state rather
than blocking invoice creation.

If the info event advertises `notifications` with `payment_received`, the app
may subscribe to notifications. Polling `lookup_invoice` remains required as
the deterministic fallback.

## Connector Interface

`0008` should implement a typed client-side connector with this shape:

```ts
type NwcConnectionStatus =
  | "missing"
  | "checking"
  | "ready"
  | "unsupported"
  | "unauthorized"
  | "relay_unreachable"
  | "error";

type NwcInvoiceState = "pending" | "settled" | "expired" | "failed";

interface NwcWalletConnector {
  connect(connectionString: string): Promise<NwcConnectionStatus>;
  disconnect(): Promise<void>;
  getCapabilities(): Promise<Set<string>>;
  getBalance(): Promise<{ balanceMsats: number }>;
  createInvoice(input: {
    amountMsats: number;
    description: string;
    expirySeconds?: number;
    metadata?: Record<string, unknown>;
  }): Promise<{
    invoice: string;
    paymentHash: string;
    amountMsats: number;
    expiresAt?: number;
  }>;
  lookupInvoice(input: {
    invoice?: string;
    paymentHash?: string;
  }): Promise<{
    state: NwcInvoiceState;
    settledAt?: number;
    preimage?: string;
  }>;
}
```

The connector should expose NWC protocol errors without leaking secrets. Map
known error codes such as `UNAUTHORIZED`, `RESTRICTED`, `NOT_IMPLEMENTED`,
`UNSUPPORTED_ENCRYPTION`, `RATE_LIMITED`, and `NOT_FOUND` into user-facing
states.

## Split-Payment Semantics

For each participant, create a distinct incoming invoice with:

- amount in millisatoshis.
- a deterministic description that includes the split identifier and
  participant index.
- metadata that does not include private personal data.

The app should advance the active invoice only after `lookup_invoice` returns a
settled incoming invoice or after a trusted `payment_received` notification is
confirmed by lookup.

## Test Routing

Focused connector tests belong to `0008`. They should cover:

- invalid connection strings.
- missing required capabilities.
- relay connection failure.
- authorization failure.
- `make_invoice` failure.
- `lookup_invoice` pending, settled, expired, and not-found states.
- `get_balance` unavailable behavior.

Regtest wallet coverage belongs to `0006` and `0007`.

## Mainnet Transport

Mainnet NWC relay access is implemented by `src/services/nwcNostrTransport.ts`
and wired for browser use by `src/services/nwcMainnet.ts`. The transport uses
the user-provided NWC URI only in client-side storage, signs Nostr requests with
the app-specific NWC credential, encrypts request and response payloads with
NIP-04, reads wallet info events from kind `13194`, sends requests as kind
`23194`, and waits for kind `23195` responses.

Automated coverage stays on regtest by default. Maintainer-run mainnet checks
are documented in `docs/nwc-mainnet-smoke-test.md` and must not commit wallet
connection strings or payment details.
